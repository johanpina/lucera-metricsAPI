Lucera Metrics API

API REST de solo lectura que alimenta el dashboard del cliente con datos reales del chatbot pediátrico. Campos, valores y rutas en inglés.

Base URL
https://lucera-metrics-877493497677.us-central1.run.app
Auth
JWT · access + refresh
Formato
JSON · UTF-8 · CORS
⚠️ Cambio de contrato

Esta versión usa campos y rutas en inglés. El mockData.ts original está en español → renombra tus tipos con la tabla de mapeo (find-replace directo).

Autenticación

JWT con dos tokens. El login devuelve un access_token (corto, 2 h) y un refresh_token (largo, 30 días). Manda el access_token en Authorization: Bearer en cada /api/*. Cuando caduque responde 401 · Token expired → pide uno nuevo con /auth/refresh sin volver a pedir contraseña.

post/auth/login → { access_token, refresh_token, user } · público
Copiar
// request  { "email":"admin@lucera.pa", "password":"Lucera2026!" }
// response
{
  "access_token":"eyJhbGci…",   // úsalo en Authorization: Bearer
  "refresh_token":"eyJhbGci…",  // guárdalo para renovar el access
  "token_type":"Bearer", "expires_in":7200,
  "user":{ "email":"…", "name":"Admin Técnico", "role":"Admin" }
}
post/auth/refresh → { access_token, expires_in } · público
Copiar
// request  { "refresh_token":"eyJhbGci…" }
// response { "access_token":"eyJhbGci…", "token_type":"Bearer", "expires_in":7200 }
// si el refresh caducó (30 días) → 401 → vuelve a hacer login

Portal del acudiente

Login separado para que un acudiente entre a ver solo sus propios datos. Se autentica con teléfono + contraseña (no email) y recibe un token con scope=portal que no puede tocar los endpoints de admin (403) — y viceversa.

post/auth/guardian/login → { access_token, refresh_token, guardian } · público
Copiar
// request  { "phone":"+57 301 5384517", "password":"MiPortal2026!" }
// response { "access_token":"eyJ…", "refresh_token":"eyJ…",
             "guardian":{ "id":"…", "name":"Johan Piña" } }
// El teléfono admite espacios/+, se normaliza a dígitos. Refresh: mismo /auth/refresh.

Registro: el acudiente fija SU contraseña (recomendado)

Un acudiente no tiene contraseña al inicio (su password_hash queda vacío, no puede entrar). El flujo previsto: el bot le manda por WhatsApp un link firmado al formulario de registro del dashboard, donde él mismo establece su contraseña.

post/api/guardians/{id}/portal-link → { token, url, expiresInHours } · token admin
get/portal/register/{token} → precarga el form · público
post/portal/register → fija la clave + activa · público
Copiar
// 1) el bot (o admin) obtiene el link firmado y lo envía por WhatsApp
//    url = {PORTAL_REGISTER_URL}?token=…  (JWT HS256, exp 72h, claims sub=guardianId, typ=register)
// 2) el form valida el token para precargar nombre/teléfono:
GET /portal/register/{token} → { "guardianId", "name", "phone", "email", "hasPassword" }
// 3) el acudiente envía su contraseña:
POST /portal/register  { "token":"eyJ…", "password":"MiClave123", "email":"opcional" }
// token vencido → 410 · inválido → 400 · clave <6 → 422

El bot puede emitir el token él mismo (JWT HS256 con el secreto compartido PORTAL_TOKEN_SECRET), sin llamar al API. PORTAL_REGISTER_URL apunta al form de Mauro.

Alternativa: el admin fija la clave directo

post/api/guardians/{id}/portal-password → { ok, id } · token admin
Copiar
{ "password":"MiPortal2026!" }   // mín. 6 caracteres; PBKDF2

Acudientes ya existentes

Los acudientes creados antes del portal no tienen contraseña (no hay clave por defecto: sería inseguro). Para saber quién ya la tiene, cada Guardian trae portalEnabled (true/false). Para onboardearlos en lote, genera sus links de registro:

get/api/portal-links → envelope de { guardianId, name, phone, hasPassword, token, url } · token admin
Copiar
// ?only_missing=true (def) → solo los que aún NO tienen clave · ?page ?page_limit
GET /api/portal-links?only_missing=true
// el bot/dashboard envía cada `url` por WhatsApp; el acudiente fija su clave.

Seguridad

El token del portal lleva gid (id del acudiente) y scope=portal. Cada endpoint filtra por ese gid: un acudiente nunca ve datos de otro. Los tokens de admin y de portal no son intercambiables (403 cruzado).

Crear / modificar / eliminar

post/api/guardians→ 201 el Guardian creado
get/api/guardians/{id}→ un Guardian
patch/api/guardians/{id}→ el Guardian actualizado
del/api/guardians/{id}→ { deleted, id }
Copiar
// POST — crea el acudiente (y su cuenta de usuario). name, phone y email obligatorios.
{ "name":"Ana Muschett", "phone":"+507 6234-2370", "email":"ana@…",
  "relationship":"mother",      // mother|father|guardian|grandparent
  "country":"Panamá", "city":"Ciudad de Panamá", "province":"Panamá",
  "status":"active",            // active|suspended|inactive (def. active)
  "plan":"premium_monthly",   // free|premium_monthly|premium_annual → registra el pago
  "insuranceId":10, "policyNumber":"POL-123" }  // seguro del acudiente (insuranceId:0 lo quita)
// email o phone duplicado → 409

// PATCH — todos los campos opcionales; solo manda lo que cambies
{ "name":"Ana M. Muschett", "email":"nueva@…", "city":"Colón",
  "relationship":"mother", "status":"suspended",
  "plan":"free", "insuranceId":12, "policyNumber":"POL-999" }

// DELETE es un borrado suave: desactiva la cuenta (deja de aparecer en la lista,
// conserva el historial de chats/pagos). → { "deleted": true, "id": "…" }
Notas de campos
  • country: editable y persistente (columna del acudiente). En acudientes antiguos sin país se infiere del prefijo del teléfono (507→Panamá, 57→Colombia).
  • plan: no es una columna; se materializa como un pago confirmado. premium_* registra el pago; free anula los pagos confirmados previos.
  • insuranceId · policyNumber: el seguro es del acudiente (lo natural: la póliza suele ser del padre). Se guarda al crear, sin depender de pacientes. Envía insuranceId:0 para quitarlo. Cada paciente conserva además su propio seguro opcional.

Conectar el dashboard (React)

src/lib/Copiar
const BASE = import.meta.env.VITE_API_URL ??
  "https://lucera-metrics-877493497677.us-central1.run.app";

export async function login(email: string, password: string) {
  const r = await fetch(`${BASE}/auth/login`, { method:"POST",
    headers:{"Content-Type":"application/json"}, body: JSON.stringify({ email, password }) });
  if (!r.ok) throw new Error("Wrong email or password");
  const { access_token, refresh_token, user } = await r.json();
  localStorage.setItem("lucera_at", access_token);
  localStorage.setItem("lucera_rt", refresh_token);
  return user;   // { email, name, role }
}
async function refresh(): Promise<boolean> {   // renueva el access sin re-login
  const rt = localStorage.getItem("lucera_rt");
  const r = await fetch(`${BASE}/auth/refresh`, { method:"POST",
    headers:{"Content-Type":"application/json"}, body: JSON.stringify({ refresh_token: rt }) });
  if (!r.ok) return false;
  localStorage.setItem("lucera_at", (await r.json()).access_token);
  return true;
}
export async function api<T>(path: string): Promise<T> {
  const hit = () => fetch(`${BASE}${path}`,
    { headers:{ Authorization:`Bearer ${localStorage.getItem("lucera_at")}` } });
  let r = await hit();
  if (r.status === 401 && await refresh()) r = await hit();  // reintenta 1 vez
  if (!r.ok) throw new Error(`Error ${r.status}`);
  return r.json();
}
// helper para listas paginadas → devuelve solo items[]
export const list = <T>(p: string, page=1, n=20) =>
  api<{ items:T[] }>(`${p}?page=${page}&page_limit=${n}`).then(d => d.items);
import { useQuery } from "@tanstack/react-query";
import { list } from "@/lib/api";
import type { Guardian } from "@/lib/types";

export function useGuardians(page=1) {
  return useQuery({ queryKey:["guardians", page],
    queryFn:() => list<Guardian>("/api/guardians", page) });   // list() ya extrae items[]
}
